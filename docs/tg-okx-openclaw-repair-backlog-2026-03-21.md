## 一、当前最重要的待修问题

### 1. 新增 `public_web` 频道时，会把频道历史消息当成“最新新消息”执行

#### 现象
- 新增频道后，系统会把该频道页面上原本已经存在的旧消息，当成新消息进入执行链。
- 用户实际观察到：明明只是新加了一个频道，但频道之前发过的内容也被执行了。

#### 当前判断的根因
- `public_web` 轮询首次接入时，没有“基线游标 / bootstrap baseline”。
- 当前逻辑只看内存里的 `_public_web_state[(channel.id, message_id)]` 是否存在：
  - 不存在 → 直接当成 `new`
  - 存在且 hash 不同 → 当成 `edit`
- 新加频道时，这个 state 对该频道是空的，所以页面上当前可见的帖子会被整批当成新消息。

#### 相关代码
- `src/tg_okx_auto_trade/telegram.py`
  - `_poll_public_web_channels(...)`
  - `_normalize_public_web_post(...)`

#### 风险
- 不只是“新加频道”会中招。
- 当前 `public_web` 状态主要在内存里，如果 runtime 重启，也存在把当前页面可见消息重新视为新消息的风险。

#### 期望修复方向
- 新频道第一次接入时，先建立基线，不执行当前页面可见历史帖。
- 后续只从“接入之后新增”的消息开始发出 `new`。
- 最好把该基线状态持久化，避免 runtime 重启后再次重放历史消息。

#### 优先级
- **P0 / 高优先级**

### 2. 管理类消息（止盈/止损/调保护/平一半等）状态语义不对

#### 现象
最近日志里，管理类消息会被 AI 识别成“管理/更新消息”，但最终落到运行态时，状态却表现成：
- `RISK_REJECTED`
- `Signal not executable`

例如：
- `止盈：74500`
- `比特币更改止盈到75000`
- `止盈2400`

AI 的 reason 实际已经明确写成：
- `Management/update message ... rather than giving a fresh trade entry`

也就是说，AI 本身知道这不是“新开仓信号”，但最终状态展示却像“风控拒绝/执行失败”。

#### 当前判断的根因
- 这类消息当前仍落在“不可执行”分支上，最终被标成 `RISK_REJECTED`。
- 语义层没有把“管理更新类消息”和“真正的风险拒绝”区分开。

#### 期望修复方向
- 对管理类消息单独落状态，不要再显示成 `RISK_REJECTED`。
- 更合适的方向包括：
  - `IGNORED`
  - `MANAGEMENT_UPDATE`
  - 或者更明确的 `PROTECTION_UPDATE_SKIPPED`
- 用户和 operator topic 看到的文案，也应明确写成：
  - 这是管理更新消息
  - 不是新开仓信号
  - 当前是忽略了，还是因为能力未实现而跳过了

#### 优先级
- **P0 / 高优先级**

---

### 3. 止盈 / 止损设置类消息当前没有真正走“保护更新执行链”

#### 现象
像下面这类消息：
- `止盈：74500`
- `比特币更改止盈到75000`
- `止损：xxxxx`

目前不会真正把保护参数写进已持仓/已挂单对象，也不会形成真实的 protection update 执行。

#### 当前判断的根因
- 现有链路更偏向：
  - 新开仓
  - 平仓
  - 撤单
  - 忽略
- 对“已有仓位 / 已有计划单的保护参数更新”，缺少明确的执行分支。

#### 期望修复方向
- 为管理类保护消息补一条明确能力：
  - `update_protection`
- 至少要能在 demo / 本地执行路径里正确落地保护状态。
- 若当前不支持真实交易所侧保护单更新，也应该至少：
  - 在本地 runtime 状态里更新保护信息
  - 在日志 / topic / Web 中明确体现“保护已更新”或“仅本地记录”

#### 优先级
- **P1 / 高优先级但次于历史消息误执行**

### 4. 管理类多句消息会被逐句独立处理，缺少“同一组信号”的合并语义

#### 现象
当前频道如果连续发：
- `比特币-市价多`
- `止盈：74500`
- `止损：xxxxx`
- `求稳的可以平一半`
- `比特币出局`

系统会把它们当成多条独立消息逐条处理，而不是理解成“同一组交易管理上下文”。

#### 当前影响
- 会导致执行日志看起来像：
  - 开多
  - 又来一条风控拒绝
  - 又来一条忽略
  - 又来一条减仓
  - 又来一条平仓
- 对 operator 来说，非常像系统在“误判/乱判”。

#### 期望修复方向
- 在当前 `public_web-first` 前提下，至少考虑：
  - 更好地利用最近上下文，把紧邻的管理类消息挂到最近一次同标的开仓上
  - 对“明显是后续管理句”的消息，别再按新交易信号那套状态展示
- 这不一定要求一步做到“消息组装引擎”，但至少要减少误导性的状态流。

#### 优先级
- **P1**

### 5. `public_web` 首次接入与重启恢复的边界行为需要重新定义并验证

#### 现象
从当前代码看：
- 首次接入新频道时没有 baseline
- runtime 重启后，内存态 `_public_web_state` 会丢
- 现有行为可能对“当前页面上可见消息”再次做 `new` 判定

#### 期望修复方向
- 明确以下场景的产品定义：
  1. 新增频道后的第一轮轮询
  2. runtime 重启后的第一轮轮询
  3. 手动 `reconcile` 后的行为
- 这三个场景都要有一致、可解释、可测试的语义。

#### 优先级
- **P1**

### 6. AI 前的上下文传递过弱，只是把最近几条原文拼给 AI

#### 现象
- 当前系统并不是完全没有给 AI 上下文。
- runtime 确实会把：
  - 当前持仓 `positions`
  - 最近几条同 chat 消息 `recent_messages`
  一起传给 AI。
- 但这层上下文仍然太弱，更像“把最近几条原文顺手附上”，而不是把它们整理成一组结构化交易上下文。

#### 当前判断的根因
- 当前 `Runtime._run_pipeline(...)` 只会取最近消息：
  - `self.storage.recent_messages(limit=5, ...)`
- 然后 `OpenClawAI._build_prompt(...)` 只是把这些 recent messages 直接塞进 prompt 的 `Recent channel context` 区块。
- 这意味着 AI 拿到的是“最近 5 条原文”，而不是：
  - 当前活跃标的
  - 最近一次有效开仓
  - 当前仓位状态
  - 当前消息与上一条的关系（新开仓 / 更新保护 / 减仓 / 平仓 / 撤单）
  这类结构化上下文。

#### 当前影响
- 连续多句本应属于同一组交易管理动作的消息，会被当成多条独立消息处理。
- AI 需要临场猜“这句是否在承接上一句”，导致：
  - 管理类消息更容易被误落状态
  - 同组交易的更新动作难以稳定绑定到上一条开仓信号上

#### 相关代码
- `src/tg_okx_auto_trade/runtime.py`
  - `_run_pipeline(...)`
- `src/tg_okx_auto_trade/ai.py`
  - `_build_prompt(...)`

#### 期望修复方向
- 在 AI 之前增加一层“上下文整理/组装”，而不是只拼接 recent messages。
- 至少要把下面这些内容结构化后再传给 AI：
  - 当前活跃标的
  - 最近一次有效开仓/加仓/减仓/平仓动作
  - 当前持仓快照
  - 最近一组同标的管理消息
  - 当前消息相对上一条消息的关系
- 让 AI 处理的是“已整理好的交易上下文”，而不是“原文堆叠”。

#### 优先级
- **P1**

### 7. 当前风控层主要是参数校验，不是交易状态风控

#### 现象
- 当前所谓“风控拒绝”，很多时候并不是真正的交易风控，而只是：
  - action / side / symbol / leverage / market_type 等合法性检查
  - paused / live_disabled / close_only / manual_confirmation / duplicate 判断
  - `intent.executable=false` / `action=ignore` 时直接落成 `RISK_REJECTED`
- 用户实际体验会变成：
  - 该拦的没拦（例如历史消息重放）
  - 不该拦的在拦（例如管理类消息被打成风控拒绝）

#### 当前判断的根因
- `RiskEngine.evaluate(...)` 当前做的更多是“执行前合法性检查”，而不是“交易语义风控 / 状态机风控”。
- 它缺少对下面这些情况的专门处理：
  - 新频道首轮历史消息误执行
  - 管理类消息不是新交易，不应走普通风险拒绝语义
  - 当前是否已有仓位 / 是否已有挂单 / 本次 action 是否与当前状态冲突
  - 同一频道短时间内连续多句是否属于同一组信号包

#### 当前影响
- 风控层在 operator 视角里显得“不像风控”，而像“格式校验 + 杂项拒绝器”。
- 很多真正该由状态机 / 上下文层处理的问题，最后被错误地显示为 `RISK_REJECTED`。

#### 相关代码
- `src/tg_okx_auto_trade/risk.py`
  - `RiskEngine.evaluate(...)`

#### 期望修复方向
- 把风险层分成至少两类：
  1. **执行合法性检查**（当前已有）
  2. **交易状态风控 / 上下文风控**（当前明显不足）
- 新增更像交易系统的检查项，例如：
  - 历史消息基线保护
  - 管理类消息分类保护
  - 已有仓位 / 订单 / 本次 action 冲突检查
  - update_protection / cancel / reduce / close 的状态语义分流
  - 同组消息节流 / 合并 / 状态机约束

#### 优先级
- **P1**

### 8. Web 交互反馈缺失，而且大量反馈仍是英文 `alert()`

#### 现象
- Web 上很多交互在执行后：
  - 要么没有明显反馈
  - 要么直接弹一个浏览器默认提示框
- 当前很多失败反馈直接是：
  - `alert(err.message)`
- 这样会导致：
  - 没有统一的页面内反馈区
  - 用户经常只看到一个突兀弹窗
  - 一旦后端返回英文错误，就直接把英文弹给用户

#### 当前判断的根因
- 当前前端交互大多仍是“动作完成后 `alert(...)` 收口”。
- 还没有统一的：
  - 页内成功/失败提示条
  - 中文化的用户提示映射
  - 更适合 operator 的反馈区/反馈样式
- 即使这轮已经把页面结构和布局重构过，交互反馈机制本身仍然比较原始。

#### 当前影响
- 用户体验上会觉得：
  - 点了没反应
  - 或者突然弹个框
  - 而且文案还不一定是中文
- 这和前面已经做的“operator-facing 默认中文”要求不一致。

#### 相关代码
- `src/tg_okx_auto_trade/web.py`
  - 当前仍有多处 `alert(err.message)` / `alert(result.detail)`

#### 期望修复方向
- 统一引入页内反馈机制：
  - 成功提示
  - 错误提示
  - 操作中状态
- 不再依赖零散的浏览器 `alert()` 作为主反馈方式。
- 对常见错误/结果文案做中文映射，至少保证 operator-facing 默认中文。
- 对关键动作（pause / resume / reconcile / topic-test / channel upsert / remove / inject-message / close positions）提供一致反馈体验。

#### 优先级
- **P1**

### 9. Web 上很多可改配置并没有真正开放可编辑入口

#### 现象
- 用户在 Web 控制台里看到一些配置，但实际上很多关键配置不能直接改。
- 你刚点名的例子是：
  - **OKX API 设置**
- 这会造成一种很差的落差：
  - 页面像控制台
  - 但很多关键配置还是只能去改文件/环境变量

#### 当前判断的根因
- 这轮 Web 重构更多是在做页面结构、布局和已有表单整理。
- 当前 Web 真正开放编辑的主要还是：
  - 交易模式部分参数
  - AI 配置部分参数
  - Telegram / topic 参数
  - Channels
- 对 OKX 凭据 / API 相关配置，当前没有真正成型的编辑入口，也没有一套清楚的“能改什么 / 为什么不能改什么 / 改完如何落盘”的交互设计。

#### 当前影响
- 用户会自然预期“既然是控制台，就应该能改关键配置”。
- 但实际很多配置没有 Web 编辑入口，会让控制台显得半成品。

#### 期望修复方向
- 明确把“可在 Web 修改的配置”补全成一套一致能力。
- 至少要明确处理 OKX 相关配置：
  - `enabled`
  - `use_demo`
  - `api_key`
  - `api_secret`
  - `passphrase`
  - `*_env`
- 如果出于安全考虑不适合直接改明文，也至少要：
  - 提供清楚的 UI 说明
  - 提供 env key 级别的配置入口
  - 明确写清哪些值走 `.env`，哪些值走 config

#### 优先级
- **P1**

### 10. AI 配置页交互和语义不清楚，当前设计“很迷”

#### 现象
- 用户直接反馈：
  - **“这个 AI 配置特别迷”**
- 说明当前 AI 配置页虽然有字段，但：
  - 字段关系不清楚
  - 默认值 / 生效方式不清楚
  - operator 不容易一眼看明白“改这个会影响什么”

#### 当前判断的根因
- 当前 AI 配置仍偏“配置文件字段直出”的思路，而不是面向 operator 的信息架构。
- 例如这些东西在 UI 里容易混在一起：
  - provider
  - model
  - thinking
  - timeout_seconds
  - system_prompt
- 但对用户来说，真正关心的通常是：
  - 现在到底用哪个 AI
  - 失败时会不会 fallback
  - 这项改动会影响解析效果还是只是推理强度
  - 哪些是推荐保持默认，哪些是常改项

#### 当前影响
- AI 配置页虽“能改”，但可理解性不足。
- 用户会觉得这块不是一个可放心操作的控制面，而像把底层字段直接摊出来。

#### 期望修复方向
- 重构 AI 配置页的信息架构，至少做到：
  - 常用项与高级项分组
  - 当前生效值更清楚
  - 各字段作用有简短中文说明
  - 默认推荐项和高风险项区分开
- 如果存在 fallback / parser_source / provider_error 等运行态概念，也应在 UI 上解释清楚，而不是只在日志里体现。

#### 优先级
- **P1**

### 11. 群组话题里的 runtime / trade / risk 广播文案没有统一中文化

#### 现象
- 用户明确要求：
  - 群组话题里的日志/回报要默认中文
- 但当前实际表现仍有大量英文，例如：
  - `[risk] ...`
  - `[execution:error] ...`
  - `[trade:observe] ...`
  - `[trade:demo] ...`
- 即使 Web 控制面已经大面积中文化，群组话题里的运行广播仍然是一半中文、一半英文。

#### 当前判断的根因
- 话题发送链本身并不会做本地化，它只是把收到的字符串原样发出去：
  - `TopicLogger.send(text)`
- 当前真正往 topic 发文案的，是 runtime 里的 `_send_topic_update(...)` 调用点。
- 而这些调用点里，很多模板仍直接写死英文，例如：
  - 风控拒绝：`[risk] {risk.code}: {risk.reason}`
  - 执行失败：`[execution:error] {action} {symbol} -> {exc}`
  - 执行成功：`[trade:{mode}] {action} {symbol} -> {result.status} (...)`
  - observe-only：`[trade:observe] ... -> observed`
- 其中 `risk.reason`、`action`、`mode`、`status`、异常 `exc` 也常常直接是英文原值。

#### 当前影响
- operator 在群组话题里看到的运行回报不统一，容易觉得：
  - Web 是中文
  - topic 却还是英文
- 这和“operator-facing 默认中文”的要求冲突。

#### 相关代码
- `src/tg_okx_auto_trade/runtime.py`
  - `_send_topic_update(...)` 的各个调用点
- `src/tg_okx_auto_trade/topic_logger.py`
  - `send(...)` 只是原样发送，不负责翻译

#### 期望修复方向
- 统一把 runtime / trade / risk / execution 这几类 topic 回报走中文模板。
- 至少要补齐这些映射：
  - `action`
  - `mode`
  - `status`
  - 常见 `risk.reason`
  - 常见执行异常原因
- 目标是：
  - 群组话题里的运行日志默认中文
  - 而不是只有 Web 页面中文

#### 优先级
- **P1**

### 12. 实盘准备项不只是更换 API，还缺一整套 live-ready 能力

#### 现象
- 用户明确指出：
  - 这个系统不只是为模拟准备的，还要为实盘准备
  - 理论上不是只换成实盘 API 就完事
- 当前代码里确实存在多处“demo-only / live 禁止 / 实盘能力缺失”的明显痕迹。

#### 当前判断的根因
- 当前项目是按 demo 路线收口的，很多地方明确写死了 demo-only 语义：
  - `risk.py` 会直接拒绝 `live`
  - `okx.py` 会直接拒绝 non-demo 实盘执行
  - `runtime` / `direct-use` / `wiring` 中仍有明显 demo 口径
- 同时，哪怕在 demo 路径里，仍有一些能力本身就没补完整：
  - `update_protection` 在真实 OKX 路径不支持
  - 首次接入 baseline 没补
  - 风控还不是交易状态风控
  - 当前状态同步仍不够像真实交易系统

#### 当前影响
- 如果只是把 API key/base 切成实盘，而不补其他能力：
  - 很容易出现“链路能下单，但整个系统并不具备 live-ready 安全性”的情况
- 也就是说，当前不是“只差换 key”，而是“还缺一整套 live-ready 收口项”。

#### 相关代码
- `src/tg_okx_auto_trade/risk.py`
  - `mode == live` 直接拒绝
- `src/tg_okx_auto_trade/okx.py`
  - non-demo 直接拒绝
  - 真实 OKX 路径仍缺 `update_protection`
- `src/tg_okx_auto_trade/runtime.py`
  - `wiring_summary()` / `usage_paths()` / `direct_use` 仍有明显 demo-only 口径

#### 期望修复方向
把“实盘准备”单独作为一组明确能力收口，而不是混成一句“以后可以切 live”。至少包括：

1. **解除 live 硬拦截的正确路径**
   - 不是直接删保护，而是设计成明确可控的 live enable 条件

2. **真实状态同步**
   - 真实仓位
   - 真实订单
   - 真实保护单
   - 不能长期只靠本地 `local_expected`

3. **保护更新链补齐**
   - `update_protection`
   - 多 TP / 多 SL / trailing 等能力在真实 OKX 路径要有清晰支持边界

4. **实盘安全闸**
   - kill switch
   - close-only
   - 单笔限额
   - 单日限额
   - 单频道限额
   - 异常自动停机
   - 更明确的 owner/operator 确认机制

5. **更强的交易状态风控**
   - 历史消息 baseline
   - 仓位冲突
   - 已挂单冲突
   - 同组消息约束
   - 重复执行抑制

6. **影子模式 / shadow mode / 预演模式**
   - 在上实盘前，先让真实信号走“解析 + 风控 + 拟执行 + 日志回报”，但不真正下单

7. **live 配置分层**
   - demo config 和 live config 分开
   - 避免在同一个配置里来回切换造成误操作

8. **实盘 operator 提示**
   - Web / topic / 小 Claw 在 live 下要明确标红或显著提示当前是实盘

#### 优先级
- **P1 / 作为后续阶段的独立工作流**

## 五、建议补的测试

### 针对 `public_web` baseline
- 新加频道后第一次轮询：
  - 页面已有旧帖，不应全部发成 `new`
- runtime 重启后第一次轮询：
  - 当前可见帖不应被重放成新消息

### 针对管理类消息状态
- `止盈：74500` / `比特币更改止盈到75000`
  - 不应落成 `RISK_REJECTED`
- `止损：xxxxx`
  - 若暂不支持执行，也应落成明确的管理类状态

### 针对 protection update
- 已有 BTC 多仓后发送 `止盈：74500`
  - 应产生 `update_protection` 或至少更新本地 protection 状态

## 五、当前结论

这轮真正要优先修的，不是“BTC 语义理解错了”，而是：

1. **`public_web` 首次接入没有 baseline，导致历史消息误执行**
2. **管理类消息的状态语义和执行分支不对**

这两个问题不修，后面 operator 看到的执行链还会继续显得“像在乱判”。
