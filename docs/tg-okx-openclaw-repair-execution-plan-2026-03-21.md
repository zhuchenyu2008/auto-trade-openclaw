这轮不是继续做零散热修，而是按一个明确顺序，把当前项目从“能跑 demo 主链路”推进到：

- 自动采集链语义更正确

## 3. 分阶段修复顺序

### Phase 0：文档与执行基线

目标：
- 把 backlog 固化
- 把执行计划固化
- 恢复 5 分钟 watchdog

完成标志：

- 执行计划文档存在
- watchdog 已恢复

---

### Phase 1：采集链与消息语义纠偏（最高优先级）

#### 1.1 `public_web` 新增频道 baseline
修复：
- 新增 `public_web` 频道时，不再把历史消息当作新消息执行
- 明确首次接入、重启恢复、reconcile 三种场景的语义

#### 1.2 管理类消息状态语义
修复：
- `止盈/止损/调保护/平一半` 这类消息，不再落成 `RISK_REJECTED`
- 区分：
  - 真风险拒绝
  - 管理更新消息
  - 暂未执行但可识别消息

#### 1.3 上下文整理增强
修复：
- 不再只是把 recent messages 原文拼给 AI
- 先整理出结构化上下文，再交给 AI

完成标志：
- 新增频道不会重放历史帖
- 管理类消息状态语义不再误导
- AI prompt 前已有更像样的上下文组织

---

### Phase 2：保护更新与交易状态风控

#### 2.1 `update_protection` 真正落地
修复：
- `止盈/止损/调保护`
- 单 TP / 多 TP / SL / trailing
- 至少本地路径完整可落地

#### 2.2 交易状态风控
修复：
- 不再只做参数校验
- 增加：
  - 仓位冲突检查
  - 已挂单冲突检查
  - 同组消息状态机约束
  - 重复消息 / 历史消息防护

完成标志：
- 管理类交易消息不再只是“ignore 或 reject”
- 风控更接近真实交易状态风控

---

### Phase 3：Web / Topic / 小 Claw operator 体验修复

#### 3.1 Web 反馈机制
修复：
- 不再大量依赖英文 `alert()`
- 引入统一页内反馈
- operator-facing 默认中文

#### 3.2 Web 可改配置补齐
修复：
- 把该改的关键配置真正开放出来
- 尤其：
  - OKX 相关配置
  - env/config 关系说明

#### 3.3 AI 配置页重构
修复：
- 常用项 / 高级项分层
- 生效值更清楚
- 中文说明更直白

#### 3.4 topic/runtime 广播中文化
修复：
- risk / trade / execution / observe 等广播默认中文
- action / status / reason / exception 做中文映射

完成标志：
- Web 与 topic 都更像面向 operator 的真实控制面
- 不再一边中文一边英文

---

### Phase 4：live-ready 准备项（不是直接开 live）

目标：
- 把“为实盘准备”单独收口，而不是一句“以后切 live 就行”

重点包括：
- live 开关路径重新设计
- 真实仓位 / 订单 / 保护单同步
- 实盘安全闸
- 影子模式 / shadow mode
- live config 分层
- live 场景下的显著提示

完成标志：
- 至少形成一套明确的 live-ready 差距清单和修复结果
- 而不是只会把 demo key 换成实盘 key
